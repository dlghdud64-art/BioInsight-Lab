* Status: 🗂️ 백로그 (비전, 미착수)
* Last Updated: 2026-06-03 (rev2 — 발송 진화 경로 A→온보딩→B 추가)
* 착수 조건: 현재 대기열(§11.335b·§11.336) + vitest install + §11.337 잔여 완료 후
* 성격: 비전 트랙 — 아이디어 합의됨, 착수 시 각 항목을 정식 SPEC으로 분할

이 문서는 여러 대화에서 발전한 입고/COA 아이디어를 흩어지기 전에 한 장으로 모은 로드맵입니다. 각 트랙은 착수 시점에 Phase breakdown을 갖춘 개별 SPEC(§11.34x)으로 전개합니다.

## 0. 한 줄 요약
연구소 입고를, "물건 들어왔다 확인"에서 → 발주↔회신↔입고↔품질(COA)이 PO·LOT으로 묶인 폐루프 + 검색 가능한 품질 아카이브로 끌어올린다. 입력은 OCR로 읽는 게 아니라 애초에 구조화된 형태로 받는다.

## 1. 핵심 원칙 (전 트랙 공통)
* 입력 채널을 통제 안으로: OCR로 종이를 읽는 건 fallback. 메인은 공급사가 우리 스키마로 직접 회신(link-based).
* 검증 게이트 불변: 입고 확정(재고·LOT mutation)은 항상 사람 승인 후. 회신/변환/COA는 "검증 대기 입고안"일 뿐 truth 아님. (§11.336 "승인 전 입고 mutation 0건" 원칙 계승)
* canonical truth 보호: 외부 데이터(회신·COA)는 derived/제안. 사람이 승인해야 canonical 승격.
* 시스템은 스크리닝, 판정은 사람: COA 교차검증은 1차 대조까지. 적합/부적합 최종 판정은 QC 담당.
* DB 부담 분리: 메타데이터는 DB(가벼움), 원본 파일(PDF/스캔)은 오브젝트 스토리지 + DB엔 링크만. (PDF를 DB BLOB로 넣지 않음)

## 2. 트랙 구성 & 의존성
```
§11.348-A  발주→회신→입고 폐루프 (link-based)        [기반]
   │
   ├─ §11.348-FALLBACK  외부문서 정규화 (QR/외부QR/OCR)  [보조, 회신 안 하는 공급사용]
   │
   └─ §11.348-B  COA
        ├─ B-1  COA 메타데이터 저장 + 조회 (아카이브)    [가벼움, 선행 가능]
        └─ B-2  COA 교차검증 (IQC)                      [품목 규격 ontology 선행 필수]
                  └─ (선행) 품목 규격 ontology           [별도 큰 트랙]
```

### §11.348-SEND — 견적/발주 요청 발송 (진화 경로 A → 온보딩 → B) · 폐루프의 입구
현재: 견적 요청서 PDF 다운로드 → 사람이 공급사 메일에 수동 첨부. "자동화인데 수동 첨부"는 모순 → 발송 자동화 필요. 단 **발송 명의(누구 이름으로 나가나)**에 따라 책임·사업모델이 갈리므로 단계적 진화로 설계.

* 1단계 — A: 유저(연구소) 명의 자동 발송 (현 설계 대상)
   * 시스템이 견적 요청서 생성 → 연구소 명의로 자동 발송, 유저는 "확인 후 일괄 발송"만(비가역 외부 발송이라 사람 확인 게이트 필수).
   * 받는 공급사 입장 기존과 동일(연구소가 보낸 것). 책임·관계·신뢰 모두 연구소에 잔류. LabAxis = 발송 주체 아닌 도구.
   * 전제(2번): 시스템 메일 발송 인프라 유무 확인 필요 — 유저 명의 발송(OAuth) vs LabAxis 서버 발송+reply-to 연구소. repo 확인 대상.
* 2단계 — 공급사 인지/온보딩 (다리)
   * 견적 받은 공급사에게 "LabAxis로 회신하면 편하다"(회신 링크, §11.348-A) 노출 → 공급사가 한 명씩 LabAxis를 인지·등록.
   * 호영님 비전: "추후 공급사도 관리 개념으로 한 곳만 보면 된다."
* 3단계 — B: LabAxis 채널 + 공급사 관리 (양면 플랫폼) (비전)
   * 공급사가 충분히 등록·동의하면 LabAxis가 정당한 거래 채널 → 공급사 관리·양면 플랫폼. 호영님이 본 "큰 거".
   * 책임 정정: 내역(audit log)이 남는 것 ≠ 면책. 명의가 책임을 진다 — LabAxis 명의 발송은 1차 책임이 LabAxis. 공급사 동의가 B의 전제. 동의 없는 B(모르는 채널 발송)는 스팸·책임·신뢰(3번 SPF/DKIM) 위험.
   * 즉 B는 "기능"이 아니라 사업모델 결정 — 공급사 동의 온보딩이 선행돼야 안전.

핵심: A로 시작(안전) → 공급사 온보딩(다리) → B 도달(양면 플랫폼). B를 포기가 아니라 안전하게 도달하는 순서.

미해결(확인 대상):
* (2번) 메일 발송 인프라가 repo에 있는지 — A의 선행. sandbox repo 확인 가능.
* (3번) 대량 발송 신뢰도(SPF/DKIM) — B로 갈수록 중요, 지금은 메모.

### §11.348-A — 발주→회신→입고 폐루프 (link-based) · 기반
* 발주 메일에 PO별 고유 회신 링크 → 공급사가 그 PO에 매핑된 폼에서 품목·수량 확정 + LOT·납기·단가 입력 → 회신이 이미 우리 스키마로 도착.
* 회신 = "검증 대기 입고안" → 사람 승인 → 입고 확정 + PO 매칭.
* 입고 현장: 확정 데이터를 QR로 출력 → 스캔만.
* 발송은 §11.348-SEND — A 단계(유저 명의 자동발송)가 이 회신 폐루프의 입구.

### §11.348-FALLBACK — 외부문서 정규화 · 보조
* 회신 안 하는 공급사용. QR/외부QR/OCR 입력 → 단일 내부 스키마로 변환.
* 신뢰도 등급: LabAxis QR(최상) > 외부QR(매핑 필요) > OCR(검증 강하게).
* 공급사별 매핑 규칙을 데이터로 축적(쓸수록 정확).
* A의 메인 경로가 닫히면 우선순위 낮음 — 폐기 아님, 보조로 상존.

### §11.348-B-1 — COA 메타데이터 저장 + 조회 (아카이브) · 가벼움·선행 가능
* COA를 LOT·품목에 묶어 저장(메타데이터 DB + 파일 스토리지 링크).
* LOT으로 즉시 조회 / 품목별 COA 이력 / 일탈·OOS 조사 시 역추적.
* 품목 규격 ontology 없이도 성립 — "받아서 묶어 저장 → 검색"만으로 현업 가치(종이 COA "못 찾음" 해결).
* 현업 니즈 명확("이전 시약/제품 COA 찾을 때 은근 있음") → B-2보다 먼저 와도 되는 독립 가치.

### §11.348-B-2 — COA 교차검증 (IQC) · 고급
* 회신 시 COA 항목별 실측치 입력(PDF 첨부는 증빙·fallback, OCR 최소화).
* 품목 규격(시험항목·기준치)과 자동 대조 → 적합 / 부적합 / 확인필요 스크리닝.
* 부적합 원자재를 입고 확정 전 차단.
* 선행 필수: 품목 규격 ontology (없으면 대조 대상이 없어 B-2 불가).

### (선행 전제) 품목 규격 ontology
* 품목별 시험항목·기준치를 담는 구조. B-2의 필수 전제이자 별도 큰 트랙.

## 3. 권장 착수 순서 (무게순)
1. §11.348-A (기반: 구조화 회신으로 입고 폐루프)
2. §11.348-B-1 (COA 아카이브 — 가볍고 현업 니즈 즉시 충족, 규격 ontology 불필요)
3. 품목 규격 ontology (B-2 선행)
4. §11.348-B-2 (COA 교차검증 IQC)
5. §11.348-FALLBACK (정규화 — A 보완, 상시 보조)

주: B-1(아카이브)은 현업 니즈가 분명하고 가벼워서, A 직후 또는 병행도 가능. 교차검증(B-2)만 규격 ontology를 기다림.
