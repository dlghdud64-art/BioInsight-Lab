## 0. 핵심 재정의
라벨은 "품목 정보 인쇄"가 아니라 **그 lot의 고유 물리 ID**. 라벨의 QR/바코드 = 재고 추적 앵커.
```
[라벨 인쇄]                [모바일 재고 스캔]          [재고 차감]
lot ID를 QR에 인코딩    →   QR 스캔 → lot 식별      →   사용량 입력 → canonical 재고 차감 + 감사
(재고 관리 화면)            (재고 스캔 화면, §11.349)    (재고 mutation)
```
세 조각이 **같은 lot ID 키**로 묶여야 폐루프가 닫힘. **QR payload(무엇을 인코딩하느냐)가 설계의 전부** — 라벨 디자인·규격은 물리적 디테일.
## 1. 현재 상태 (화면 확인 + 가설)
- **라벨 QR 미출력:** 모달 preview엔 QR 보이는데 실제 인쇄 출력(`about:blank`)엔 바코드(`||||`)만 있고 QR 없음 → **preview-actual 불일치, dead toggle.** 재고 차감 관점에선 단순 버그가 아니라 **폐루프 시작점 끊김.**
- **규격 불일치:** 목록(폼텍 3100/3101/3102/3104, DYMO 11354)이 실물/실사용과 안 맞음(호영님). 실사용 = 폼텍 3104, 단 **한 연구소 사례** — 하드코딩 금지.
- **모바일 스캔 화면:** 존재(앞서 확인). 단 카메라 중지 dead button(§11.349) 선행 필요.
- **재고 차감 연동:** **아마 백지(미구현) — Phase 0 코드 확인 필수.** (호영님: "아마 백지일 것, 코드 확인 필요")
## 2. 핵심 원칙
- **QR payload = lot 고유 ID.** 단순 텍스트(품목명) ❌ → 차감 대상을 특정하는 키 ✅.
- **라벨 규격 = 데이터, 하드코딩 금지.** 3104는 한 연구소 사례. 프리셋 + 커스텀 입력 + 연구소별 기본값. (재고 정책 §11.346과 같은 철학)
- **차감은 canonical mutation + 사람 확인.** 스캔=식별, 차감=사용량 입력 후 실행. front-only 차감 금지.
- **preview = actual.** "QR 포함" 토글이 실제 출력에 반드시 반영(dead toggle 금지).
## 3. 조각별 트랙 (착수 시 SPEC 분할)
### §11.355-A — QR payload 표준 (선행, 다른 게 다 의존)
- QR에 인코딩할 lot 식별자 스키마 정의(inventory lot ID 등). 스캔 시 그 lot을 즉시 특정 가능.
- 모바일 스캔 디코드와 동일 표준 공유.
### §11.355-B — 라벨 발행 (실규격 + QR 실출력)
- **QR 실제 출력 버그 수정** (preview만이 아니라 인쇄 HTML에 QR 생성).
- **규격 데이터화:** 프리셋(폼텍 3104 등 실검증된 치수) + 커스텀 입력(가로×세로·시트당 칸·여백) + 연구소별 기본값 설정.
- **실규격 대조:** 목록의 폼텍/DYMO 치수가 실제 제품과 맞는지 검증(임의값이면 정정).
### §11.355-C — 모바일 스캔 → lot 식별
- QR 디코드 → §11.355-A 표준으로 정확한 lot 조회. (§11.349 카메라 lifecycle 선행)
### §11.355-D — 재고 차감 mutation (단편화 해소)
- **정정(호영님 확인):** 차감은 **완전 백지가 아님** — **글로벌 스캐너엔 "사용/차감" CTA가 이미 구현됨.** 누락된 건 **전용 스캔 페이지(§11.349)** 쪽 → page-per-feature 단편화.
- 따라서 작업 = 차감 로직 신규 구현이 아니라, **기존 글로벌 스캐너의 차감 CTA를 전용 스캔 페이지에도 연결**(같은 기능, 같은 mutation 재사용). 단편화 해소가 핵심.
- 스캔된 lot에 사용량 입력 → canonical 재고 차감 + 감사 기록. 사람 확인 게이트.
