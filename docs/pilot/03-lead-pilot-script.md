# Ops Console V1 — Lead 파일럿 스크립트

> 역할: Lead (`lead@demo.lab`, APPROVER)
> 소요 시간: 약 35분
> 목적: 승인/거버넌스/개선 흐름 + 에스컬레이션 검증

---

## 사전 준비

- [ ] 시드 데이터 7건 투입 완료 확인
- [ ] `lead@demo.lab` 계정 로그인
- [ ] 기록 템플릿 (04-feedback-template.md) 준비
- [ ] Operator 파일럿이 먼저 수행되었으면 좋음 (필수는 아님)

---

## Step 1. 콘솔 진입 — 운영 큐 탭 (5분)

**경로**: 대시보드 → 운영 콘솔 → `운영 큐` 탭

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 1-1 | Lead 관점에서 전체 큐가 보이는가 | 7건 모두 표시 | |
| 1-2 | `approvals_needed` 그룹이 강조되는가 | 승인 대기 그룹 상단 노출 | |
| 1-3 | 자신에게 배정된 항목이 구별되는가 | owner 컬럼에 본인 표시 | |
| 1-4 | `shouldActorAct = true`인 항목의 CTA가 활성화되는가 | 승인/인수 CTA 활성 | |

### 수행 동작
1. 전체 큐 스캔 — Lead가 조치해야 할 항목 식별
2. 자신에게 배정된 항목 수 확인

---

## Step 2. 구매 승인 처리 (5분)

**대상**: 시나리오 2 — "FBS 구매 승인 요청"

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 2-1 | `approvals_needed` 그룹에 위치하는가 | 해당 그룹 내 | |
| 2-2 | 금액 (₩385,000)이 row에 보이는가 | 금액 표시 | |
| 2-3 | 긴급 사유가 보이는가 | "재고 안전 수준 미만" | |
| 2-4 | 승인 CTA가 Primary로 표시되는가 | "승인" 버튼 강조 | |
| 2-5 | 상세 패널에서 견적 내역을 확인할 수 있는가 | 제품명, 수량, 금액 | |

### 수행 동작
1. Row 클릭 → 상세 패널 열기
2. 견적 내역, 요청 사유, 재고 상황 확인
3. "승인" CTA 클릭
4. 승인 후 상태 변경 확인 (PENDING → APPROVED)
5. 해당 아이템이 `approvals_needed`에서 빠지는지 확인

---

## Step 3. 인수인계 수락 (5분)

**대상**: 시나리오 5 — "DMEM 견적 검토 인수인계"

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 3-1 | `stalled_handoffs` 그룹에 위치하는가 | 해당 그룹 내 | |
| 3-2 | 인수인계 보낸 사람(operator)이 표시되는가 | handoffFrom 표시 | |
| 3-3 | 인수인계 후 경과 시간이 보이는가 | "3일 전" | |
| 3-4 | "인수 수락" CTA가 있는가 | Primary CTA 활성 | |

### 수행 동작
1. "인수 수락" CTA 클릭
2. AssignmentState: `HANDED_OFF` → `HANDED_OFF_ACCEPTED` 전환 확인
3. 수락 후 해당 아이템의 owner가 자신으로 변경되었는지 확인

---

## Step 4. 일일 검토 — 에스컬레이션 (5분)

**경로**: 콘솔 → `일일 검토` 탭

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 4-1 | `needs_lead_intervention` 카테고리가 보이는가 | Lead 전용 카테고리 표시 | |
| 4-2 | 차단 7일 초과 아이템이 에스컬레이션 대상으로 표시되는가 | 시나리오 4 포함 | |
| 4-3 | 에스컬레이션 액션이 선택 가능한가 | escalate_blocked 등 | |
| 4-4 | carry-over 사유가 표시되는가 | 이월 사유 명시 | |

### 수행 동작
1. `needs_lead_intervention` 카테고리 내 항목 확인
2. 시나리오 4 (차단 아이템)에 대해 에스컬레이션 액션 수행
3. 에스컬레이션 후 상태 변화 확인

---

## Step 5. 거버넌스 탭 (5분)

**경로**: 콘솔 → `거버넌스` 탭

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 5-1 | Cadence 테이블이 표시되는가 | 주기별 단계 표시 | |
| 5-2 | SLA 상태 테이블이 표시되는가 | SLA 초과/준수 현황 | |
| 5-3 | Lead Intervention 트리거가 표시되는가 | 개입 기준 목록 | |
| 5-4 | Governance Signals가 보이는가 | 신호 그리드 | |

### 수행 동작
1. 각 섹션 스크롤하며 데이터 유무 확인
2. SLA 초과 항목이 시드 데이터와 일치하는지 확인
3. 거버넌스 탭의 정보가 의사결정에 도움이 되는지 메모

---

## Step 6. 개선 탭 (5분)

**경로**: 콘솔 → `개선` 탭

**대상**: 시나리오 6 — "입고 병목 개선"

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 6-1 | Summary strip에 병목 현황이 보이는가 | 통계 표시 | |
| 6-2 | 병목 테이블에 RECEIVING 항목이 있는가 | 입고 병목 row | |
| 6-3 | 현재 메트릭 vs 목표 메트릭이 표시되는가 | 4.2일 → 2.0일 | |
| 6-4 | 개선 상태 (IN_PROGRESS)가 표시되는가 | 상태 badge | |

### 수행 동작
1. 병목 분류별 현황 확인
2. 개선 진행 중인 항목의 상세 확인
3. 이 화면이 Lead 의사결정에 유용한지 메모

---

## Step 7. 기록 정리 (5분)

1. 위 표에서 "실제 결과" 열 작성
2. `04-feedback-template.md`에 기록, 특히:
   - **"Lead로서 오늘 가장 먼저 봐야 할 것이 바로 보였는가?"**
   - **"승인 판단에 필요한 정보가 row/패널에 충분했는가?"**
   - **"거버넌스/개선 탭이 실제로 유용했는가, 아니면 안 볼 것 같은가?"**
   - **"에스컬레이션 흐름이 자연스러웠는가?"**
   - **"Operator의 작업 상황이 한눈에 파악되었는가?"**
