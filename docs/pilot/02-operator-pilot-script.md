# Ops Console V1 — Operator 파일럿 스크립트

> 역할: Operator (`operator@demo.lab`, REQUESTER)
> 소요 시간: 약 30분
> 목적: 운영 큐 중심의 일상 업무 흐름 검증

---

## 사전 준비

- [ ] 시드 데이터 7건 투입 완료 확인
- [ ] `operator@demo.lab` 계정 로그인
- [ ] 기록 템플릿 (04-feedback-template.md) 준비

---

## Step 1. 콘솔 진입 — 운영 큐 탭 (5분)

**경로**: 대시보드 → 운영 콘솔 → `운영 큐` 탭

### 확인 항목

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 1-1 | 콘솔 진입 시 첫 화면이 `운영 큐`인가 | 운영 큐 탭 활성 | |
| 1-2 | Summary strip에 전체 건수 / 긴급 건수가 보이는가 | 숫자 표시 | |
| 1-3 | 그룹이 우선순위 순서대로 정렬되어 있는가 | urgent_blockers → approvals_needed → ... | |
| 1-4 | 각 row에 title, state, owner, age, CTA가 보이는가 | 7개 항목 모두 row 형태 | |

### 수행 동작
1. 그룹 목록을 위에서 아래로 스크롤하며 전체 구조 파악
2. 각 그룹의 row 수 확인

---

## Step 2. 긴급 차단 아이템 처리 (5분)

**대상**: 시나리오 4 — "Anti-CD3 항체 재주문 차단"

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 2-1 | `urgent_blockers` 그룹 최상단에 위치하는가 | 첫 번째 그룹 | |
| 2-2 | 빨간 severity border가 표시되는가 | border-l 빨간색 | |
| 2-3 | 차단 사유가 row에 보이는가 | "벤더 미응답" | |
| 2-4 | age가 표시되는가 | "7일" | |
| 2-5 | Primary CTA가 1개만 강조되는가 | "벤더 후속 조치" 버튼 | |

### 수행 동작
1. 해당 row의 Primary CTA 클릭
2. 상세 패널(Sheet) 열림 확인
3. 상세 패널에서 차단 사유, SLA 초과 정보, 연결된 견적 확인
4. 패널 닫기

---

## Step 3. 입고 대기 아이템 확인 (5분)

**대상**: 시나리오 3 — "Trypsin 입고 대기"

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 3-1 | `receiving_restock` 그룹에 위치하는가 | 해당 그룹 내 | |
| 3-2 | 배송 지연이 표시되는가 | 예상일 초과 표시 | |
| 3-3 | "입고 확인" CTA가 있는가 | Primary CTA 표시 | |
| 3-4 | 연결된 Order 정보가 표시되는가 | Order 상세 링크 | |

### 수행 동작
1. Row 클릭 → 상세 패널 열기
2. 주문일, 예상 배송일, 현재 age 확인
3. "입고 확인" CTA 동작 확인

---

## Step 4. 인수인계 상태 확인 (5분)

**대상**: 시나리오 5 — "DMEM 견적 검토 인수인계"

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 4-1 | `stalled_handoffs` 그룹에 위치하는가 | 해당 그룹 내 | |
| 4-2 | 인수인계 상태가 표시되는가 | "인수인계 대기" | |
| 4-3 | 인수인계 대상(lead)이 표시되는가 | lead 이름/ID | |
| 4-4 | Operator에게 CTA가 없거나 제한적인가 | 수락 CTA 없음 (lead만 가능) | |

### 수행 동작
1. Row에서 인수인계 상태 읽기
2. 상세 패널에서 handoff 이력 확인
3. Operator 입장에서 할 수 있는 동작 확인 (없어야 정상)

---

## Step 5. 일일 검토 탭 확인 (5분)

**경로**: 콘솔 → `일일 검토` 탭

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 5-1 | 탭 전환이 동작하는가 | 일일 검토 화면 표시 | |
| 5-2 | 카테고리별 분류가 보이는가 | urgent_now, overdue_owned 등 | |
| 5-3 | carry-over 표시가 있는 항목이 있는가 | 이월 badge | |
| 5-4 | 최근 완료 항목이 하단에 있는가 | recently_resolved 카테고리 | |

### 수행 동작
1. 각 카테고리 확인
2. 시나리오 7 (최근 완료) 항목이 `recently_resolved`에 있는지 확인
3. 시나리오 4 (차단)가 `urgent_now`에 있는지 확인

---

## Step 6. 완료 항목 확인 (3분)

**대상**: 시나리오 7 — "피펫팁 벤더 응답 완료"

| # | 확인 | 예상 결과 | 실제 결과 |
|---|------|-----------|-----------|
| 6-1 | `recently_resolved` 그룹에 위치하는가 | 하단 그룹 | |
| 6-2 | 완료 시점이 표시되는가 | "6시간 전" | |
| 6-3 | CTA가 없거나 네비게이션만 있는가 | "상세 보기"만 | |

---

## Step 7. 기록 정리 (2분)

1. 위 표에서 "실제 결과" 열 작성
2. 운영 흐름에서 막히거나 혼란스러운 점 → `04-feedback-template.md`에 기록
3. 특히 아래를 중심으로 메모:
   - **"이 화면에서 다음에 뭘 해야 하는지 바로 알았는가?"**
   - **"정보가 충분히 보이는가, 상세 패널을 열어야만 알 수 있는 건 없는가?"**
   - **"CTA 클릭 후 기대한 곳으로 이동했는가?"**
